Map & Reduce

Description

This project is my implementation of the MIT distributed systems MapReduce lab.
MapReduce is a programming model for processing large data sets by splitting work into two stages:
map functions generate intermediate key/value pairs, and reduce functions combine all values for each key.
The interesting part of the lab was not just running map and reduce code, but coordinating many workers,
tracking task progress, and recovering when a worker stops responding.

Technical Concepts

The implementation is written in Go, which fits the lab well because goroutines and RPC make it natural
to build a coordinator that talks to multiple worker processes. The coordinator acts like a small state machine:
it starts in the map phase, waits for all map tasks to finish, moves to the reduce phase, and finally marks the job done.

During the map phase, the coordinator creates one task for each input file and assigns available tasks to idle workers.
Each worker reads its input file, runs the user-defined Map() function, partitions the intermediate key/value pairs
into reduce buckets, and writes those intermediate files to disk. During the reduce phase, workers read the intermediate
files for their partition, sort and group by key, run Reduce(), and write the final output files.

The coordinator tracks every task as idle, in progress, or completed. If a worker takes too long or exits early,
the coordinator treats the task as unfinished and makes it available again. This retry behavior is what lets the system
pass the crash and early-exit tests in the lab test suite.
